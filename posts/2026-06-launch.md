# Shipping gfix M2: an AI merge layer that explains instead of guessing

*Alpha launch post. The architectural sections describe the M2 design choices; the dogfood section is the receipt — two real sessions, eight findings, one cross-machine rerere replay, and a security finding caught by the red team that was written to find it.*

---

## The wedge

Every coding agent now produces branches in parallel — Claude Code, Cursor, Aider, Codex, Gemini CLI. The merge is where parallel work goes to die. Incumbents either hide the merge behind a vendor cloud (Replit), bolt it onto a single-agent assistant inside a website (Copilot), or hand it back to GitHub after isolating with worktrees (Cursor). `gfix`'s bet: the merge resolver should be local, private, and speak the one protocol every agent already speaks — MCP. Not an orchestrator. A resolver.

## The deterministic floor (M1 recap)

M1 shipped the unglamorous half that matters: text merge, then Mergiraf (invoked as a subprocess — Mergiraf is GPL-3.0 and we never link it), then a structured conflict list for everything the floor can't settle. No AI. The floor never regresses and never hallucinates. Everything M2 adds sits on top of this floor; nothing replaces it.

**This earned its place in dogfood the first time we tried to test the AI path.** Session 1's first fixture was two branches editing the same TypeScript function signature — one making it `async`, one changing a parameter type. The intent was to force the AI ceiling. Mergiraf's tree-sitter TypeScript grammar treated both as well-formed AST surgeries and merged them cleanly with no conflict at all. We had to pivot to a `.txt` file (outside Mergiraf's allowlist) to force the conflict downstream. The structural-merge floor doing more than expected is a product win — it shrinks the AI-call budget on real codebases.

## The AI ceiling — sampling-first, honest about ~50%

M2's headline is the AI suggestion path. The choice that matters: **sampling-first**. When the MCP host advertises a sampling capability, gfix asks the host's LLM — the user's model, the user's bill, the user's privacy posture. BYO-key (Anthropic / OpenAI / Gemini / Ollama) is the opt-in fallback; Ollama is the fully-offline path.

The honest part: independent research (merde.ai) puts LLM resolution of complex semantic conflicts around 50%. gfix doesn't pretend otherwise. The AI's job is not "auto-fix everything" — it's "explain what each side did and propose a resolution the human can accept in one glance." A suggestion is never auto-applied. The value is the stack, not the model.

**Empirical wrinkle from dogfood: Claude Code — our canary MCP host — does not advertise `sampling/createMessage` today.** The original capability matrix framed BYO-key as the *fallback*. In practice, for Claude Code users in 2026, BYO-key is the *primary* AI path, not the fallback. The new framing in [capability-matrix.md](../capability-matrix.md) leads with BYO-key and treats sampling as a future-host affordance. This was finding #2 from the dogfood log — a doc-positioning bug that became the headline reframe of the launch.

## rerere as the amortization layer

The next agent on the next branch never re-asks the same question. Any accepted resolution is hashed by its conflict triple (`file_path`, `base_oid`, `ours_oid`, `theirs_oid`) via blake3 and written to `refs/gitfix/rerere/<hash>` as a synthetic commit. The next time that exact conflict appears — across branches, machines, teammates who pushed the ref — gfix replays the accepted answer before the AI path is reached. The 50%-accurate AI call is paid once; the answer is reused forever. That's what makes a mediocre ceiling economically fine.

**This is the load-bearing demo, and session 2 proved it.** More on the numbers below.

## Audit refs as the provenance story

Every applied merge writes an immutable synthetic commit at `refs/gitfix/audit/<merge_id>` whose tree holds the full final merge state as `audit.json`. Visible in `git log --all`, pushable independently (`git push origin refs/gitfix/audit/*`), readable on another machine with one `git show`. When an agent's merge is questioned three weeks later, the answer is a 30-second ref read, not archaeology. gfix never pushes — provenance travels when the human chooses to.

## Red-teaming the framing — and finding a real bug

M2 Slice 7a built a prompt-injection corpus. The thesis is precise: gfix's safety property is not "the model can't be jailbroken" — it's "a jailbroken model can only produce file content, never actions." gfix writes `proposed` bytes to a file and stores `rationale` verbatim; it never executes or interprets model output.

Building the corpus surfaced a genuine vulnerability: the untrusted conflict body was concatenated raw into the prompt, so a literal `</conflict>` token in hostile content could walk the model out of the "untrusted" wrapper the system prompt relies on. The original design reasoned "we don't parse the wrapper back, so strict correctness isn't required" — true for our code, false for the model, which is the entire audience of the wrapper. We fixed it in the same slice (defang any wrapper-close sequence in content) and the corpus now regression-guards it.

Worth saying plainly: **the red-team work paid for itself by finding the bug it was written to find.** Finding and fixing your own injection bug before shipping is the credibility signal. We're not sanding that off.

## Dogfood: I merged gfix into gfix, and here's what happened

Two sessions. Session 1 was supposed to be 90 minutes; it took 3 calendar days because of setup friction. Session 2 was supposed to be 30 minutes; it took 5. The gap between the two is the entire pitch.

### Session 1 — three days for what was scheduled as ninety minutes

**Goal:** drive the AI suggestion path end-to-end on a real conflict; verify `AiSuggestionAccepted` in the audit log.

**Outcome:** worked, on attempt 3, after three calendar days. Eight findings documented; one of them caused an emergency mid-dogfood feature add.

**Timeline:**

| Day | What happened |
|---|---|
| Fri 2026-05-22 10:00-11:30 IST | `merge_preview` worked on attempt 3 fixture; BYO-key setup hit walls |
| Sat 2026-05-23 | BYO-key troubleshooting continued; `keys.toml` path bug surfaced; three keys leaked in chat; Gemini added as emergency Slice 4.5 |
| Sun 2026-05-24 | Resumed with working byok-gemini; `conflict_get` + `conflict_resolve` + `merge_apply` completed; audit ref written |

**The three friction nodes that ate the 90 minutes:**

1. **Two fixtures merged structurally before reaching the AI path.** Attempt 1 was TypeScript signature edits — Mergiraf resolved it. Attempt 2 was JSON — Mergiraf resolved it. Attempt 3 was `.txt`, outside Mergiraf's allowlist; that was the one that forced the conflict to fall through to the text floor and require an AI suggestion. This is *good news* (Mergiraf earns its place), but it cost an iteration cycle to discover.

2. **The keys.toml path on macOS diverges from XDG.** The docs said `~/.config/gitfix/keys.toml`. The `directories` Rust crate, on macOS, resolves to `~/Library/Application Support/dev.gitfix.gitfix/keys.toml`. The first three attempts to edit `keys.toml` targeted the wrong file; gfix read none of them silently. No diagnostic. Doc fix landed for alpha.1 (both paths documented); proper fix (honor `XDG_CONFIG_HOME` cross-platform) lands alpha.2.

3. **Three OpenAI API keys leaked in chat during setup.** Twice via copy-paste into the wrong input field; once when my own sed redaction script ran unquoted on macOS (BSD `sed` doesn't support `\s` in character classes). Each key was revoked within seconds. None caused real harm — the keys were quota-walled on a brand-new account with no billing attached — but the lesson is generalizable: the setup story now hard-routes API keys through `$EDITOR` + `keys.toml`, never the command line. The `-e API_KEY=...` shape is removed from all docs. The keys-leak risk was a product-shape bug, not a "user error" — designing onboarding that makes accidental disclosure easy is the developer's fault, not the user's.

**The emergency mid-dogfood feature add.** All three documented BYO-key paths — Anthropic, OpenAI, Ollama — hit blocking friction during the Saturday setup window. Anthropic billing wasn't configured. OpenAI walled the new account's free tier. Ollama's macOS arm64 compile is a multi-hour first install. The unblock was Gemini Flash's free tier — added as a fourth provider in a two-hour unplanned Slice 4.5 commit (`bdf202a`), included in the same push as Slice 7a. The dogfood then immediately exercised the new path end-to-end. This is the cleanest example of dogfood-driven development in the M2 timeline: the session forced a real feature add, which the session then validated.

**What actually shipped at the end of session 1:**

- **merge_id:** `m_2026-05-22T10-25-45Z_c2bc70`
- **HEAD oid:** `acbad3af5b69b60d27927b63c4555ffca5fed798` (3-parent octopus: main + feat/a2 + feat/b2)
- **Audit ref:** `refs/gitfix/audit/m_2026-05-22T10-25-45Z_c2bc70`
- **Decision log (excerpt):**

```json
"decisions": [{
  "timestamp": "2026-05-23T19:15:40.439653Z",
  "conflict_id": "c_2a3d7",
  "kind": "ai_suggestion_accepted",
  "actor": "ai-agent:claude-code"
}]
```

`actor: "ai-agent:claude-code"` is load-bearing: Slice 3's MCP client-info extraction correctly attributed the decision to the agent that drove it.

**Gemini suggestion (excerpt) on the conflict:**

```json
{
  "proposed": "Status: in beta. We aim for stability by Q4. See roadmap.",
  "confidence": 0.10,
  "rationale": "Arbitrary choice between two contradictory updates. Neither branch has clear semantic priority.",
  "source": "byok-gemini"
}
```

`confidence: 0.10` is notably low. Gemini refused to fake certainty on a genuine ambiguity. **That is the right behavior.** Pretending high confidence on a coin flip would be worse than the coin flip — at least the coin flip is honest about being random.

### The eight findings (the full list)

Documented in full in the source repo's `docs/dogfood/m2-w1.md`; summarized here:

1. **Mergiraf handles TypeScript signature changes structurally (product win).** Capability matrix updated.
2. **Claude Code does not advertise `sampling/createMessage`.** Capability matrix reframe: BYO-key is primary.
3. **`keys.toml` silent parse on missing `[providers.<name>]` section header.** Setup docs now require the header explicitly. `gfix doctor` candidate for alpha.2.
4. **`claude mcp add -e VAR=val NAME` variadic consumed the server name.** Setup docs now show `--` separator explicitly.
5. **`claude mcp add` defaults to `--scope local`.** Setup docs now use `--scope user` everywhere.
6. **macOS `keys.toml` path diverges from XDG.** Setup docs now document both paths. `gfix --version` prints the resolved path. Proper fix at alpha.2.
7. **OpenAI free tier walls new accounts immediately in 2026.** Setup docs now route new users to Gemini Flash.
8. **Slice 4.5 (Gemini provider) was an emergency mid-dogfood add.** Informational — the retrospective's lead anecdote.

(Plus a bonus: `.gitfix/state.json` swept into the merge tree via `git add -A`. Auto-gitignore fix lands alpha.2.)

Every one of the eight is addressed in alpha.1 — most in docs, one in code (auto-gitignore deferred to alpha.2), one as a documented workaround. The findings are not hidden; they're [setup.md §6](../setup.md#6-troubleshooting). Surfacing the friction is the credibility move.

### Session 2 — five minutes wall-clock, zero AI calls

**Goal:** verify cross-machine rerere replay works. Resolution accepted in session 1 (`refs/gitfix/rerere/a2eea8...`) should auto-replay on a fresh fixture in a separate repo, with no AI call, by traveling the rerere ref through a plain bare git remote.

**Setup (5 lines of git, no gfix-specific tooling):**

```sh
# 1. Bare repo as the shared rerere ref store
git init --bare -b main /tmp/gitfix-shared-bare

# 2. From session 1's repo, push the rerere refs
cd /tmp/gitfix-dogfood-w1
git remote add shared /tmp/gitfix-shared-bare
git push shared 'refs/gitfix/rerere/*:refs/gitfix/rerere/*'

# 3. Create session 2's repo with identical fixture content (same STATUS.txt bytes as w1)
#    ... script omitted; produces blob OID c156cef3... on both feat/a2 and main ...

# 4. Fetch rerere refs into session 2's repo
git config --add remote.origin.fetch '+refs/gitfix/rerere/*:refs/gitfix/rerere/*'
git fetch origin
```

Content-addressing made this trivial: identical `(base, ours, theirs)` bytes in session 2 produced identical blob OIDs to session 1, which produced an identical rerere hash, which matched the ref name pushed from session 1.

```
session 1 STATUS.txt blob OID on feat/a2:  c156cef3d47ab495f6a006c6f7569f703b21d22e
session 2 STATUS.txt blob OID on feat/a2:  c156cef3d47ab495f6a006c6f7569f703b21d22e
```

Rerere ref confirmed present in session 2 after fetch: `refs/gitfix/rerere/a2eea82d4ae074092bb4c765636a29a441d92020e6e1d05e2fecf6d96ba555b2`.

**The session itself: 3 MCP tool calls, total wall-clock under 5 minutes.**

```
gitfix_merge_preview -> STATUS.txt in `resolved` array with "via": "rerere".
                       Empty `unresolved`. No AI suggestion generated.
                       (rerere hit before classification reached unresolved.)
gitfix_merge_apply  -> 3-parent octopus merge written. auto_approve=true.
git show refs/gitfix/audit/m_2026-05-23T20-09-27Z_cebcd8:audit.json
                     -> kind: "rerere_replay", ai_suggestions_used: 0
```

**The audit ref (real, fetchable):**

```json
{
  "audit_schema_version": 1,
  "gitfix_version": "0.1.0",
  "merge_id": "m_2026-05-23T20-09-27Z_cebcd8",
  "decisions": [{
    "kind": "rerere_replay",
    "ai_suggestions_used": 0
  }]
}
```

**The Gemini key generated Wednesday for session 2 was never used.** Rerere replay fired before any provider was queried. 0 Gemini calls, 0 OpenAI calls, 0 Ollama calls. The exposure window on that key was zero API requests.

### The quantitative anchor

- **Session 1: 3 calendar days** for one AI-driven merge resolution. Most of that is setup friction (the eight findings).
- **Session 2: 5 minutes wall-clock** for the same conflict shape, with no AI call.
- **AI call latency in session 1:** ~1-2 seconds for the Gemini suggestion.
- **rerere replay latency in session 2:** sub-100ms (local git ref lookup + blob materialization).

The 10-20x latency speedup is the demo line. The 3-days-to-5-minutes story is the credibility line — and it's only credible because the 3 days is documented honestly. **Sanding the 3 days down to "session 1 worked" would have lost the entire arc.**

This is also why "AI ceiling is ~50% and that's fine" is a defensible claim. You pay the 50% accuracy cost once. The replay is the deterministic floor reaching backwards in time to absorb the AI's win.

## Honest about ceilings — and the closed-source posture

**The AI ceiling is real, and it's the point.** ~50% on complex semantic conflicts is the merde.ai number — and that 50% is the half Mergiraf can't reach. Mergiraf is the deterministic floor that resolves the easy half; gfix is the AI + audit layer that resolves the hard half. gfix doesn't beat the model; gfix wraps the model so that the wins compound (rerere) and the losses are visible (low confidence scores, full rationale, never auto-applied). The product is the wrapper, not the model.

**The closed-source posture is a choice, not a permanent state.** `gfix` ships as a closed-source binary in alpha. The 5-line EULA lets you install on any number of machines for any purpose including commercial use; you can't redistribute the binary or reverse-engineer it. Free during alpha. Pricing decision deferred to v1.0. The closed-source decision is provisional and re-decided at v1.0 as a single block — possible endpoints include open-core, paid-team, or fully open under a permissive license. The alpha posture isn't gospel; it's "ship now, decide later, don't pretend the decision is made when it isn't."

The trust substitute for source visibility is the audit refs. Every merge gfix performs is inspectable as a plain git commit on a synthetic ref. The session 1 and session 2 merges above (`m_2026-05-22T10-25-45Z_c2bc70` and `m_2026-05-23T20-09-27Z_cebcd8`) are real artifacts you can fetch and inspect — they're not paste-job claims, they're git objects.

## What's next (M3)

M3 makes gfix something a human installs and lives in: standalone CLI dispatch (drive a merge from a shell, no MCP host needed — `gfix merge feat/a feat/b`), an opt-in jj substrate for jj users, and a thin VS Code panel that renders the merge log and turns Accept/Reject into one click — over the same MCP transport the engine already speaks. Same engine, three new front doors.

Beyond M3, the open questions are: when does the AI ceiling become a problem (which is to say: when do users complain about a specific conflict shape gfix gets wrong, often enough to deserve a fine-tune lane)? When does the team-features lane open (gfix is single-developer-first today; team features are a "if users ask, build it; don't preempt the ask" item)? And the v1.0 license decision — re-opened once alpha usage data exists.
